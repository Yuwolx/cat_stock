# 주식 AI 분석 프로그램 진행 계획 및 파일 구조

작성일: 2026년 5월 29일

## 1. 목표

이 프로젝트의 1차 목표는 아래입니다.

- 비개발자도 브라우저에서 바로 사용할 수 있는 웹 화면 제공
- 버튼 클릭으로 데이터 수집 실행
- 수집 결과를 AI 채팅창에 붙여넣기 좋은 텍스트로 출력
- 별도 AI API 연동 없이 수동 복사용 워크플로우 유지
- 복잡한 자동화 대신 먼저 수동 실행 버전부터 안정적으로 완성

즉, 첫 단계에서는 "매일 자동 분석 시스템"이 아니라  
"사용자가 웹에서 버튼만 눌러 텍스트를 생성하는 리서치 도구"를 만드는 것이 목표입니다.

---

## 2. 왜 이 방향으로 가는가

현재 조건상 가장 중요한 것은 개발 속도보다도 사용 편의성입니다.

- 사용자는 비개발자이므로 VS Code 실행 방식은 불편함
- Airflow, 서버 스케줄링, 복잡한 배포는 초기 단계에서 과함
- 먼저 수동 실행 버전으로 실제 사용성을 검증하는 것이 안전함
- 버튼 기반 웹앱이면 사용자는 URL 접속 후 클릭만 하면 됨

따라서 1차 구조는 아래처럼 가져갑니다.

- 개발 언어: Python
- 웹 UI: Streamlit
- 배포: Railway
- 실행 방식: 수동 실행
- 결과물: 화면 내 텍스트 출력 + 복사/다운로드
- 1차 버전에서는 프롬프트 자동 첨부 없이 데이터 결과만 제공

---

## 3. 1차 MVP 범위

처음부터 모든 기능을 다 넣지 않고, 아래 순서로 만듭니다.

### 1단계

Module 1 시황 브리핑만 먼저 구현

포함 범위:

- 한국 지수 기본 정보
- 거래대금 상위 종목
- 외국인 / 기관 수급 일부
- 글로벌 매크로 일부
- 주요 공시 일부
- 결과 텍스트 생성
- 화면 출력
- `.txt` 다운로드

### 2단계

Module 3 개별 종목 분석 추가

포함 범위:

- 종목 입력창
- 기본 정보
- 수급
- 재무 요약
- 공시 / 뉴스 / 리포트 요약
- 결과 텍스트 생성

### 3단계

Module 2 테마 공부 추가

포함 범위:

- 테마명 입력
- 관련 종목 목록
- 뉴스 / 공시 / 리포트
- 글로벌 피어 정보
- 결과 텍스트 생성

## 4. 사용자 사용 방식

비개발자 사용자는 아래 흐름만 알면 됩니다.

1. 웹 주소 접속
2. 원하는 메뉴 선택
3. 필요한 입력값 작성
4. `생성` 버튼 클릭
5. 화면에 나온 텍스트 복사
6. 본인이 쓰는 프롬프트와 함께 ChatGPT 또는 Gemini에 붙여넣기

사용자에게 Python 실행, 터미널, 포트, 로컬 서버 개념은 노출하지 않습니다.

---

## 5. 운영 방식

초기 운영은 수동 실행 기반으로 합니다.

- 사용자가 필요할 때 웹에서 직접 실행
- 서버는 Railway에 배포
- 데이터 수집은 요청이 들어왔을 때만 수행
- 결과는 DB에 저장하지 않고 우선 화면 출력 중심으로 운영

초기에는 아래 원칙으로 단순하게 갑니다.

- 로그인 없음
- 회원관리 없음
- 관리자 화면 없음
- 자동 스케줄링 없음
- 복잡한 백엔드 분리 없음

## 6. 권장 기술 구조

### 프론트/UI

- Streamlit

이유:

- Python만으로 빠르게 화면 구성 가능
- 입력창, 버튼, 텍스트 출력, 다운로드 버튼 구현이 쉬움
- 데이터 앱 형태에 적합함

### 데이터 수집

- `requests`
- `BeautifulSoup`
- 필요 시 `pandas`
- 필요 시 `pykrx`
- 필요 시 `yfinance`
- DART OpenAPI

### 배포

- Railway

이유:

- 비개발자 사용을 위한 고정 URL 제공
- 로컬 서버를 매번 띄울 필요 없음
- 월 5달러 수준으로 운영 가능
- 추후 스케줄 작업 추가도 비교적 쉬움

## 7. 시스템 개념 구조

전체 흐름은 아래와 같습니다.

1. 사용자가 웹 화면에서 기능 선택
2. Streamlit 앱이 해당 수집 함수 실행
3. 각 데이터 소스에서 정보 수집
4. 수집 결과를 공통 포맷으로 정리
5. 최종 복사용 텍스트 생성
6. 화면에 출력
7. 사용자가 복사 또는 다운로드

핵심은 "수집"과 "텍스트 조립"을 분리하는 것입니다.

- 수집 로직: 사이트/API에서 데이터 가져오기
- 포맷 로직: 가져온 데이터를 문장/리스트로 정리
- UI 로직: 입력, 버튼, 출력만 담당

이렇게 나누면 나중에 유지보수가 쉬워집니다.

## 9. 파일별 역할

### 루트

- `app.py`
  - Streamlit 시작점
  - 메뉴 선택, 버튼 클릭, 결과 출력 담당

- `requirements.txt`
  - 배포와 실행에 필요한 패키지 목록

- `.env.example`
  - DART API 키 등 환경변수 예시

- `railway.json`
  - Railway 배포 설정

### `src/collectors/`

- 외부 사이트/API에서 원본 데이터를 가져오는 역할
- 출처별로 파일을 나누어 관리

### `src/services/`

- 여러 collector 결과를 합쳐서 하나의 분석 데이터로 만드는 역할
- UI와 수집 로직 사이의 중간 계층

### `src/formatters/`

- 데이터를 AI 입력용 텍스트로 가공
- 최종 `.txt` 형식 조립 담당

### `src/utils/`

- 공통 기능 모음
- 날짜 처리, 파일 저장, 공통 문자열 처리, 로깅 등

### `src/ui/`

- Streamlit 화면용 재사용 UI 코드
- 버튼, 입력창, 결과 출력 블록 등

### `tests/`

- 텍스트 포맷이 깨지지 않는지 확인
- 핵심 데이터 조립 로직 테스트

## 10. 1차 개발 순서

아래 순서로 만드는 것이 가장 안전합니다.

1. 프로젝트 기본 폴더 구조 생성
2. Streamlit 기본 화면 생성
3. 시황 브리핑 더미 데이터 출력
4. 실제 데이터 수집기 1개씩 연결
5. 결과 텍스트 저장 및 다운로드 기능 추가
6. Railway 배포
7. 배포 환경에서 수집 안정성 점검

중요한 점은 처음부터 크롤링을 전부 붙이지 않는 것입니다.

먼저:

- 버튼이 눌리는지
- 화면 출력이 되는지
- 텍스트 구조가 만족스러운지
- 다운로드가 잘 되는지

이걸 먼저 확인해야 합니다.

---

## 11. 배포 방식

배포 구조는 단순하게 갑니다.

1. GitHub 저장소 생성
2. Railway에 저장소 연결
3. 환경변수 등록
   - `DART_API_KEY`
4. 앱 실행 명령 설정
5. 배포 후 URL 발급

사용자는 이 URL만 사용하면 됩니다.

---

## 12. 나중 자동화 확장 방식

자동화는 1차 완료 후 별도 단계로 진행합니다.

확장 방법 예시:

- Railway cron/job으로 장 마감 후 실행
- 생성 결과를 파일로 저장
- 최근 생성 결과를 웹에서 다시 열람
- 이메일/텔레그램/노션 전송 연동

중요한 점:

지금은 자동화보다 "수동 실행 웹앱의 완성도"가 더 중요합니다.

## 14. 최종 제안

현재 기준 최선의 시작 방식은 아래입니다.

- 수동 실행 웹앱으로 시작
- Streamlit으로 빠르게 UI 구현
- Railway로 배포해서 URL 제공
- Module 1부터 완성
- 나중에 자동화는 별도 추가

이 방식이 가장 적은 비용으로,
가장 빠르게 실제 사용 가능한 형태를 만들 수 있습니다.
